## 前置准备（开工前完成）

| 序号 | 事项 | 负责 | 备注 |

## 第1周：项目搭建 + 基础框架

### 目标：能跑起来，有登录和基础页面骨架

| 任务 | 说明 | 产出 |
|------|------|------|
| 1.1 初始化 Taro 项目 | `taro init`，配置 React + TS + NutUI | 项目骨架 |
| 1.2 云开发初始化 | 创建云环境，配置数据库集合 | 云函数+数据库 |
| 1.3 数据库建表 | Product / Tour / Order / OrderItem / OrderNote / DeliveryLocation / User | 7个集合 |
| 1.4 用户登录 | 微信授权登录，角色选择（导游/供货商） | 登录流程 |
| 1.5 基础布局 | TabBar导航、页面路由、通用组件（Header/Card/Button） | UI骨架 |
| 1.6 商品数据初始化 | 把7个商品录入数据库 | 初始数据 |

**里程碑：小程序能打开，能登录，能看到商品列表**

---

## 第2周：导游端 — 下单核心流程

### 目标：导游能完整走通 建团→填白单→提交

| 任务 | 说明 | 产出 |
|------|------|------|
| 2.1 团次管理页 | 创建团次（团号/日期），团次列表，团次详情 | 3个页面 |
| 2.2 白单填写页（手动模式） | 商品列表+数量选择器，白单编号输入，合计金额计算 | 核心页面 |
| 2.3 配送方式选择 | 酒店送货/景点自提/快递到家，常用地点库 | 配送组件 |
| 2.4 订单确认+提交 | 确认页（商品/数量/金额/配送信息），提交到数据库 | 提交流程 |
| 2.5 汇总单页面 | 按团次汇总的电子白单（跟纸质白单一样的表格） | 汇总页面 |
| 2.6 订单列表+详情 | 导游查看自己的历史订单，按状态筛选 | 2个页面 |

**里程碑：导游能完整下单，能看到汇总单**

---

## 第3周：供货商端 + 订单状态流转

### 目标：供货商能接单、发货；导游能跟踪状态

| 任务 | 说明 | 产出 |
|------|------|------|
| 3.1 供货商订单列表 | 按状态分Tab（待确认/待发货/已发货/已完成），新订单高亮 | 列表页 |
| 3.2 订单详情（供货商视角） | 商品清单、配送信息、接单/拒单按钮 | 详情页 |
| 3.3 接单+拒单 | 确认接单，或拒单+填写原因 | 状态流转 |
| 3.4 标记发货 | 供货商标记已发货 | 状态流转 |
| 3.5 导游确认收货 | 导游端确认收到货 | 状态流转 |
| 3.6 订阅消息通知 | 新订单通知供货商、接单/发货通知导游 | 消息推送 |
| 3.7 订单备注留言 | 双向留言板 + 新留言通知 | 沟通功能 |
| 3.8 联系对方 | 跳转微信联系按钮 | 兜底沟通 |

**里程碑：完整订单流转跑通（下单→接单→发货→收货），双方都能用**

---

## 第4周：AI 识别 + 配送优化

### 目标：拍照识别白单，配送地址智能管理

| 任务 | 说明 | 产出 |
|------|------|------|
| 4.1 拍照组件 | 相机组件 + 拍照引导框 + 图片预处理 | 拍照页面 |
| 4.2 OCR 云函数 | 调用腾讯云表格 OCR API，解析返回结果 | 云函数 |
| 4.3 识别结果映射 | OCR 结果 → 商品名匹配 → 数量提取 → 置信度标记 | 解析逻辑 |
| 4.4 识别确认页 | 显示 AI 识别结果，标注不确定项，导游确认修改 | 确认页面 |
| 4.5 AI 大模型兜底 | OCR 失败时调用大模型识别，作为备用方案 | 云函数 |
| 4.6 常用地点管理 | 地点库 CRUD，按使用频率排序，记忆历史地点 | 地点管理 |
| 4.7 地址变更通知 | 导游修改地址后自动通知供货商 | 通知逻辑 |

**里程碑：AI 拍照识别可用，配送地址管理完善**

## 风险和应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 小程序审核不通过 | 延迟上线 | 提前了解审核规则，第5周就提交初版 |
| AI识别准确率不够 | 用户体验差 | 手动模式兜底，识别只是辅助 |
| 导游不习惯用小程序 | 推广难 | 界面尽量简单，操作步骤少于纸质流程 |
| 旅游景区信号差 | 下单失败 | 本地缓存，联网后自动同步 |
| 订阅消息限制 | 通知收不到 | 引导用户授权，备用短信通道 |
