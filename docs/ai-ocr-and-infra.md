### 两种输入方式并存

| 方式 | 适合场景 | 优点 | 缺点 |
|------|---------|------|------|
| 📸 拍照识别 | 已有纸质白单 | 快，几秒完成 | 手写潦草时可能识别错 |
| ✍️ 手动填写 | 边聊边录 / 没带白单 | 准确 | 慢一点 |

导游自由选择，两种方式最终都进入同一个确认页面。

## 二、技术方案

### OCR 识别流程

```
拍照
 ↓
图片上传到云端
 ↓
表格结构识别（找到行列）
 ↓
单元格文字识别（读数字和勾选标记）
 ↓
映射到商品（列名→商品名，单元格→数量）
 ↓
返回结构化数据
 ↓
前端展示让导游确认
```

### 技术选型

| 方案 | 说明 | 价格 | 推荐度 |
|------|------|------|--------|
| **腾讯云 OCR - 表格识别** | 专门识别表格，返回行列结构化数据 | 500次/月免费，超出0.15元/次 | ⭐⭐⭐ 推荐 |
| 微信 OCR 插件 | 小程序内置，简单文字识别 | 免费 | ⭐⭐ 只能识别文字，不识别表格结构 |
| 百度 OCR - 表格识别 | 表格识别，质量好 | 500次/天免费 | ⭐⭐⭐ 备选 |
| Claude/GPT 多模态 | 发图片让AI直接读表格 | ~0.01-0.05元/次 | ⭐⭐⭐ 准确度最高 |

**推荐方案：腾讯云表格OCR + AI大模型兜底**

```
拍照 → 腾讯云表格OCR（快，结构化）
              ↓
         识别成功？
        /        \
      是          否（模糊/歪斜）
      ↓              ↓
  返回结果      发给AI大模型识别（慢但准）
                     ↓
                 返回结果
```

## 三、技术难点

### 难点1：手写数字识别（中等难度）

**问题**：白单是手写的，字迹因人而异，"1"和"7"、"6"和"0"容易混淆

**解决**：
- 白单格式固定（商品列是固定的），只需要识别数字和勾选标记
- 加置信度标记，低于阈值的标黄让导游确认
- 多数情况数量是 1-5，范围小，容易纠正

**评估**：✅ 可解决。白单结构规范，识别难度不高

### 难点2：表格结构识别（低难度）

**问题**：需要把图片里的表格行列对应到商品

**解决**：
- 白单表头是印刷的（罗汉果、桂圆肉...），表头识别准确率高
- 表格线清晰，结构规范
- 第一次识别后可以缓存表格模板

**评估**：✅ 容易。白单是标准印刷表格，不是随意手画的

### 难点3：拍照质量（中等难度）

**问题**：户外拍照可能光线不好、歪斜、有阴影

**解决**：
- 小程序拍照时加辅助框引导对齐
- 图片预处理（自动纠偏、增强对比度）
- 识别失败时提示重拍或切换手动模式

**评估**：✅ 可解决。加拍照引导就行

### 难点4：新商品适配（低难度）

**问题**：将来商品可能增减，表头会变

**解决**：
- 商品列表从数据库读取，OCR结果和数据库商品名做模糊匹配
- 管理员加新商品时同步更新匹配规则

**评估**：✅ 容易

### 总结

| 难点 | 难度 | 是否有成熟方案 | 风险 |
|------|------|--------------|------|
| 手写数字 | 中 | 有，OCR+置信度 | 低 |
| 表格结构 | 低 | 有，腾讯云表格OCR | 低 |
| 拍照质量 | 中 | 有，引导+预处理 | 低 |
| 新商品适配 | 低 | 有，模糊匹配 | 低 |

**结论：没有硬性技术障碍，都有成熟方案。最大风险是手写太潦草导致识别率低，但有人工确认兜底，不影响使用。**

## 四、基础设施 (Infrastructure)

### 方案 A：微信云开发（推荐 MVP）

最简单，不用管服务器。

```
┌──────────────────────────────────────────────────┐
│                   微信云开发                       │
│                                                   │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐  │
│  │   云函数    │  │   云数据库  │  │   云存储    │  │
│  │ (后端逻辑)  │  │ (MongoDB)  │  │ (图片/文件) │  │
│  └────────────┘  └────────────┘  └────────────┘  │
│                                                   │
│  ┌────────────────────────────────────────────┐   │
│  │              消息推送（订阅消息）             │   │
│  └────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────┘
         ↑                              ↑
    小程序前端                      腾讯云OCR
   (Taro/React)                   (API调用)
```

| 资源 | 免费额度 | 超出价格 | 预估月用量 |
|------|---------|---------|-----------|
| 云函数调用 | 100万次/月 | 0.05元/万次 | <10万次 ✅ |
| 云数据库 | 2GB | 按需付费 | <500MB ✅ |

### 方案选择建议

```
MVP 阶段（前3个月）
  → 方案 A（微信云开发）
  → 成本：¥0
  → 理由：零运维，快速上线验证

增长阶段（用户量上来后）
  → 迁移到方案 B
  → 成本：¥100-160/月
  → 理由：更灵活，支持复杂查询和更大数据量
```
